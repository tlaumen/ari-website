Van terminal naar team: bouw je eerste AI workflow met ARI

Tom Laumenjuni 202610 min lezen

Een AI workflow is geen chatbot met extra knoppen. Het is een manier om vast te leggen waar code rekent, waar AI helpt en waar de ingenieur controle houdt.

In de vorige posts hebben we het vooral gehad over principes:

  • zet AI niet in waar deterministische code beter is
  • maak AI functies betrouwbaar met routing, context en evals
  • begin klein en verzamel data uit echt gebruik

Dat klinkt logisch. Maar hoe ziet het eruit als je het bouwt?

In dit blog maken we een eerste ARI workflow. Niet als eindproduct, maar als werkende demo. Eerst lokaal, daarna in een browseromgeving voor je team.

Het voorbeeld is een stabiliteitsberekening. Dat is bewust gekozen. De berekening zelf wil je niet door een taalmodel laten verzinnen. Die moet reproduceerbaar zijn. Maar een taalmodel kan wél helpen met een eerste rapporttekst, zolang de ingenieur die tekst controleert en accepteert.

Dat is precies het punt.

Waarom geen chat?

Jacob Lauritzen van Legora vatte het goed samen:

“The ceiling of chat interfaces isn’t a UX problem. It’s a verification problem.”

Voor civiel ingenieurs is dit direct herkenbaar.

Je kan aan een AI vragen: “maak een stabiliteitsrapport”. Dan krijg je waarschijnlijk een tekst die er goed uitziet. Maar waar is de berekening gedaan? Welke parameters zijn gebruikt? Waar heeft de ingenieur gecontroleerd? En welke versie is uiteindelijk geaccepteerd?

Met een workflow maken we die stappen expliciet.

Niet alles wordt AI. Niet alles wordt chat. De berekening blijft code. De rapporttekst mag AI zijn. De controle blijft bij de ingenieur.

Wat gaan we bouwen?

We bouwen een eenvoudige workflow voor een stabiliteitsberekening:

Grondprofiel invoeren
        ↓
Parameters invoeren
        ↓
Geometrie invoeren
        ↓
Stabiliteitsberekening uitvoeren
        ↓
Conceptrapport schrijven
        ↓
Feedback van de gebruiker verwerken

De veiligheidsfactor komt uit een open-source Pythonpakket voor stabiliteit. De rapporttekst komt uit een AI functie. De ingenieur bepaalt welke rapportversie telt.

Aan het einde pushen we de workflow naar een demo-omgeving:

uv run ari demo-push

Daarmee draait dezelfde workflow in de browser. Je collega hoeft geen Python, uv of repository te kennen. Hij opent de link, vult de invoer in, draait de berekening en beoordeelt het rapport.

Dat is de stap van terminal naar team.

Project opzetten

ARI gebruikt uv als package manager. Een nieuw project aanmaken doe je met:

uv run ari new

Voor deze demo gebruiken we naast ARI ook xslope voor de stabiliteitsberekening:

uv add ari-core xslope tabulate openai anthropic

Daarna werk je verder in workflow.py.

Een AI coding agent kan hier goed bij helpen. ARI projecten bevatten een LLM_GUIDE.md, zodat een agent de structuur van het project begrijpt. Je hoeft dus niet eerst het hele framework te leren voordat je een eerste workflow kan bouwen.

De bouwstenen

Een ARI workflow bestaat uit drie simpele onderdelen.

Een DataModel beschrijft de data die door je workflow stroomt. Denk aan grondlagen, parameters, geometrie, rekenresultaten of rapporttekst.

Een Step doet één ding. Bijvoorbeeld: vraag het grondprofiel uit, voer geometrie in, voer een berekening uit of genereer een rapport.

Een Segment is de workflow zelf: een lijst van steps in de volgorde waarin ze uitgevoerd worden.

Daarnaast kent ARI twee plekken om data op te slaan:

  • Table.PROJECT: data die bij het project hoort en bewaard moet blijven
  • Table.WORKFLOW: tijdelijke data binnen één run

Vuistregel:

Is het een deliverable of projectgegeven? Gebruik Table.PROJECT.
Is het een tussenresultaat? Gebruik Table.WORKFLOW.

Deel 1 — Een eerste stabiliteitsworkflow

We beginnen bewust eenvoudig.

Geen custom UI. Geen HTML. Geen JavaScript. Geen CPT-parser.

Alle invoer is gestructureerd. De geometrie voeren we in als maaiveldpunten. Dat is genoeg om het patroon te laten zien:

gestructureerde invoer
+
deterministische berekening
+
AI-rapport
+
menselijke feedback

We bouwen de workflow stap voor stap. Bij iedere stap voegen we alleen de code toe die op dat moment nodig is.

Stap 1 — Grondprofiel invoeren

De gebruiker voert een eenvoudig grondprofiel in.

Bijvoorbeeld:

NAP +8.0 m tot NAP +5.0 m: klei
NAP +5.0 m tot NAP +1.0 m: zand
NAP +1.0 m tot NAP -2.0 m: vaste zandlaag

In ARI maken we hier eerst getypeerde data van. De helper as_float(...) houdt de conversie uit de weg, zodat de ARI-primitives zichtbaar blijven: prompt(...), confirm(...), DataModel en ctx[...].

from __future__ import annotations

from ari_core.models import DataModel, Table
from ari_core.prompting import confirm, prompt
from ari_core.workflow.step import Product, Step, StepContext


def as_float(value: object) -> float:
    return float(str(value).replace(",", "."))


class Grondlaag(DataModel):
    bovenkant: float
    onderkant: float
    materiaal: str


class Grondprofiel(DataModel):
    lagen: list[Grondlaag]
    grondwaterstand_nap: float


class VoerGrondprofielInStep(Step):
    name = "voer_grondprofiel_in"
    requires = []
    produces = [Product(key="grondprofiel", dest=Table.PROJECT)]

    def execute(self, ctx: StepContext) -> None:
        maaiveld_nap = prompt("Wat is het maaiveldniveau (NAP)?")

        lagen: list[Grondlaag] = []
        huidige_bovenkant = as_float(maaiveld_nap)

        while True:
            laag_nr = len(lagen) + 1
            onderkant = prompt(f"Laag {laag_nr} - onderkant (NAP):")
            materiaal = prompt(f"Laag {laag_nr} - materiaal:")

            lagen.append(
                Grondlaag(
                    bovenkant=huidige_bovenkant,
                    onderkant=as_float(onderkant),
                    materiaal=str(materiaal).strip(),
                )
            )

            huidige_bovenkant = as_float(onderkant)

            doorgaan = confirm("Nog een laag toevoegen?", default=False)
            if not doorgaan:
                break

        grondwaterstand_nap = prompt("Wat is de grondwaterstand (NAP)?")

        ctx["grondprofiel"] = Grondprofiel(
            lagen=lagen,
            grondwaterstand_nap=as_float(grondwaterstand_nap),
        )

Deze step produceert grondprofiel en bewaart dat in Table.PROJECT.

Dat is logisch: het grondprofiel hoort bij het project. Dit wil je later opnieuw kunnen gebruiken.

Stap 2 — Parameters invoeren

Daarna vragen we de materiaalparameters uit per grondsoort die in het grondprofiel voorkomt.

Bijvoorbeeld:

  • volumiek gewicht
  • cohesie
  • hoek van inwendige wrijving

Dit is geen prompt die ergens verdwijnt. Het zijn expliciete projectgegevens.

from __future__ import annotations

from ari_core.models import DataModel, Table
from ari_core.prompting import prompt
from ari_core.workflow.step import Product, Requirement, Step, StepContext

from steps.voer_grondprofiel_in import Grondprofiel, as_float


class GrondsoortParameters(DataModel):
    volumiek_gewicht_kN_m3: float
    cohesie_kPa: float
    wrijvingshoek_graden: float


class MateriaalParameters(DataModel):
    per_materiaal: dict[str, GrondsoortParameters]


class VoerParametersInStep(Step):
    name = "voer_parameters_in"
    requires = [Requirement(key="grondprofiel", source=Table.PROJECT, model=Grondprofiel)]
    produces = [Product(key="materiaal_parameters", dest=Table.PROJECT)]

    def execute(self, ctx: StepContext) -> None:
        grondprofiel: Grondprofiel = ctx["grondprofiel"]
        materialen = sorted({laag.materiaal for laag in grondprofiel.lagen})

        per_materiaal: dict[str, GrondsoortParameters] = {}

        for naam in materialen:
            print(f"Voer parameters in voor materiaal: {naam}")
            gamma = prompt(f"{naam} - volumiek gewicht (kN/m3):")
            cohesie = prompt(f"{naam} - cohesie (kPa):")
            phi = prompt(f"{naam} - wrijvingshoek (graden):")

            per_materiaal[naam] = GrondsoortParameters(
                volumiek_gewicht_kN_m3=as_float(gamma),
                cohesie_kPa=as_float(cohesie),
                wrijvingshoek_graden=as_float(phi),
            )

        ctx["materiaal_parameters"] = MateriaalParameters(per_materiaal=per_materiaal)

Deze step vraagt het grondprofiel op als input. Dat is handig: je voert parameters in voor de grondsoorten die al bekend zijn.

Ook deze output gaat naar Table.PROJECT.

Stap 3 — Geometrie invoeren

Veel civieltechnische workflows beginnen met eenvoudige projectdata. In deze demo voeren we de geometrie in als puntenlijst en zetten die daarna om naar workflowdata.

De helper parse_geometriepunten(...) houdt de string-parsing buiten de step-logica. Daardoor blijft de kern zichtbaar: de gebruiker geeft input via prompt(...) en de step slaat een Geometrie model op in de context.

from __future__ import annotations

from ari_core.models import DataModel, Table
from ari_core.prompting import prompt
from ari_core.workflow.step import Product, Requirement, Step, StepContext

from steps.voer_grondprofiel_in import as_float
from steps.voer_parameters_in import MateriaalParameters


class GeometriePunt(DataModel):
    x: float
    z: float


class Geometrie(DataModel):
    punten: list[GeometriePunt]


def parse_geometriepunten(value: object) -> list[GeometriePunt]:
    punten = []
    for paar in str(value).split(";"):
        x, z = paar.split(",")
        punten.append(GeometriePunt(x=as_float(x), z=as_float(z)))

    return sorted(punten, key=lambda p: p.x)


class LaadGeometrieStep(Step):
    name = "laad_geometrie"
    requires = [Requirement(key="materiaal_parameters", source=Table.PROJECT, model=MateriaalParameters)]
    produces = [Product(key="geometrie", dest=Table.PROJECT)]

    def execute(self, ctx: StepContext) -> None:
        punten = prompt(
            "Geef maaiveldpunten als x,z-paren gescheiden door puntkomma's. "
            "Voorbeeld: 0,8.0; 10,7.5; 20,5.0"
        )

        ctx["geometrie"] = Geometrie(
            punten=parse_geometriepunten(punten)
        )

Belangrijker dan de invoervorm is het patroon: zodra de geometrie is ingevoerd, is het gewone getypeerde workflowdata.

Stap 4 — Stabiliteitsberekening uitvoeren

Nu komt de rekenkern.

De workflow gebruikt het grondprofiel, de materiaalparameters en de geometrie als input voor een open-source Pythonpakket dat de stabiliteit berekent.

De uitkomst bestaat hier uit:

  • veiligheidsfactor
  • middelpunt en straal van de maatgevende glijcirkel

Dit is bewust geen AI-stap.

De helper build_slope_data(...) zet de workflowdata om naar het formaat van de rekenkern. run_analysis(...) voert de xslope-analyse uit en geeft de maatgevende resultaten terug.

from ari_core.models import DataModel, Table
from ari_core.workflow.step import Product, Requirement, Step, StepContext

from helpers.xslope_engine import build_slope_data, run_analysis
from steps.laad_geometrie import Geometrie
from steps.voer_grondprofiel_in import Grondprofiel
from steps.voer_parameters_in import MateriaalParameters


class StabiliteitsResultaat(DataModel):
    veiligheidsfactor: float
    maatgevende_cirkel_x0: float
    maatgevende_cirkel_y0: float
    maatgevende_cirkel_r: float


class BerekenStabiliteitStep(Step):
    name = "bereken_stabiliteit"
    requires = [
        Requirement(key="grondprofiel", source=Table.PROJECT, model=Grondprofiel),
        Requirement(key="materiaal_parameters", source=Table.PROJECT, model=MateriaalParameters),
        Requirement(key="geometrie", source=Table.PROJECT, model=Geometrie),
    ]
    produces = [Product(key="stabiliteitsresultaat", dest=Table.WORKFLOW)]

    def execute(self, ctx: StepContext) -> None:
        grondprofiel: Grondprofiel = ctx["grondprofiel"]
        materiaal_parameters: MateriaalParameters = ctx["materiaal_parameters"]
        geometrie: Geometrie = ctx["geometrie"]

        slope_data = build_slope_data(
            grondprofiel=grondprofiel,
            materiaal_parameters=materiaal_parameters,
            geometrie=geometrie,
        )
        analyse = run_analysis(slope_data, num_slices=30)

        ctx["stabiliteitsresultaat"] = StabiliteitsResultaat(
            veiligheidsfactor=analyse["veiligheidsfactor"],
            maatgevende_cirkel_x0=analyse["maatgevende_cirkel_x0"],
            maatgevende_cirkel_y0=analyse["maatgevende_cirkel_y0"],
            maatgevende_cirkel_r=analyse["maatgevende_cirkel_r"],
        )

Het resultaat zetten we eerst in Table.WORKFLOW, omdat het de input is voor de rapportstap. Als je het rekenresultaat ook als deliverable wil bewaren, kun je het later ook in Table.PROJECT opslaan.

Het belangrijkste punt is dit:

AI schrijft niet de veiligheidsfactor.
De veiligheidsfactor komt uit de rekenkern.

Stap 5 — Conceptrapport genereren

Pas na de berekening gebruiken we AI.

De AI functie krijgt gestructureerde input:

  • grondprofiel
  • parameters
  • geometrie
  • berekeningsresultaat
  • eventueel projectcontext

Daarmee schrijft hij een eerste rapporttekst.

Dit is een goede plek voor AI. De taak is makkelijk te controleren. Een ingenieur kan de tekst lezen, vergelijken met de invoer en beoordelen of de conclusie klopt.

De helper genereer_rapport(...) maakt de rapporttekst op basis van de gestructureerde invoer. Later gebruiken we dezelfde helper opnieuw met feedbackpunten.

from ari_core.models import DataModel, Table
from ari_core.workflow.step import Product, Requirement, Step, StepContext

from helpers.rapport_generator import genereer_rapport
from steps.bereken_stabiliteit import StabiliteitsResultaat
from steps.laad_geometrie import Geometrie
from steps.voer_grondprofiel_in import Grondprofiel
from steps.voer_parameters_in import MateriaalParameters


class ConceptRapport(DataModel):
    titel: str
    inhoud_markdown: str


class GenereerRapportStep(Step):
    name = "genereer_rapport"
    requires = [
        Requirement(key="grondprofiel", source=Table.PROJECT, model=Grondprofiel),
        Requirement(key="materiaal_parameters", source=Table.PROJECT, model=MateriaalParameters),
        Requirement(key="geometrie", source=Table.PROJECT, model=Geometrie),
        Requirement(key="stabiliteitsresultaat", source=Table.WORKFLOW, model=StabiliteitsResultaat),
    ]
    produces = [Product(key="concept_rapport", dest=Table.WORKFLOW)]

    def execute(self, ctx: StepContext) -> None:
        grondprofiel: Grondprofiel = ctx["grondprofiel"]
        materiaal_parameters: MateriaalParameters = ctx["materiaal_parameters"]
        geometrie: Geometrie = ctx["geometrie"]
        stabiliteitsresultaat: StabiliteitsResultaat = ctx["stabiliteitsresultaat"]

        inhoud = genereer_rapport(
            grondprofiel=grondprofiel,
            materiaal_parameters=materiaal_parameters,
            geometrie=geometrie,
            stabiliteitsresultaat=stabiliteitsresultaat,
            feedback_pointers=None,
        )

        ctx["concept_rapport"] = ConceptRapport(
            titel="Concept stabiliteitsrapport",
            inhoud_markdown=inhoud,
        )

De volgorde is belangrijk:

eerst rekenen
daarna schrijven
daarna controleren

Zo voorkom je dat AI de rekenkundige waarheid gaat vervangen.

Stap 6 — Feedback vragen en geaccepteerde versie opslaan

De conceptrapportage is nog geen deliverable.

De gebruiker moet hem beoordelen.

Wat moet er beter aan deze rapporttekst?
Welke pointers geef je mee voor de finale versie?

De workflow vraagt pointers op en draait daarna dezelfde rapportfunctie opnieuw. Die aangepaste versie slaan we op als geaccepteerde rapporttekst.

from __future__ import annotations

from ari_core.models import DataModel, Table
from ari_core.prompting import prompt
from ari_core.workflow.step import Product, Requirement, Step, StepContext

from helpers.rapport_generator import genereer_rapport
from steps.bereken_stabiliteit import StabiliteitsResultaat
from steps.genereer_rapport import ConceptRapport
from steps.laad_geometrie import Geometrie
from steps.voer_grondprofiel_in import Grondprofiel
from steps.voer_parameters_in import MateriaalParameters


class RapportFeedback(DataModel):
    opmerkingen: str


class GeaccepteerdRapport(DataModel):
    inhoud_markdown: str


def parse_pointers(value: object) -> list[str]:
    return [p.strip() for p in str(value).split(";") if p.strip()]


class VraagRapportFeedbackStep(Step):
    name = "vraag_rapport_feedback"
    requires = [
        Requirement(key="grondprofiel", source=Table.PROJECT, model=Grondprofiel),
        Requirement(key="materiaal_parameters", source=Table.PROJECT, model=MateriaalParameters),
        Requirement(key="geometrie", source=Table.PROJECT, model=Geometrie),
        Requirement(key="stabiliteitsresultaat", source=Table.WORKFLOW, model=StabiliteitsResultaat),
        Requirement(key="concept_rapport", source=Table.WORKFLOW, model=ConceptRapport),
    ]
    produces = [
        Product(key="rapport_feedback", dest=Table.PROJECT),
        Product(key="geaccepteerd_rapport", dest=Table.PROJECT),
    ]

    def execute(self, ctx: StepContext) -> None:
        grondprofiel: Grondprofiel = ctx["grondprofiel"]
        materiaal_parameters: MateriaalParameters = ctx["materiaal_parameters"]
        geometrie: Geometrie = ctx["geometrie"]
        stabiliteitsresultaat: StabiliteitsResultaat = ctx["stabiliteitsresultaat"]
        concept_rapport: ConceptRapport = ctx["concept_rapport"]

        print("\n=== Concept rapport ===\n")
        print(concept_rapport.inhoud_markdown)

        opmerkingen = prompt(
            "Geef pointers voor verbetering (scheid met ';'). "
            "Voorbeeld: voeg duidelijke conclusie toe; benoem aannames expliciet"
        )

        aangepaste_rapporttekst = genereer_rapport(
            grondprofiel=grondprofiel,
            materiaal_parameters=materiaal_parameters,
            geometrie=geometrie,
            stabiliteitsresultaat=stabiliteitsresultaat,
            feedback_pointers=parse_pointers(opmerkingen),
        )

        print("\n=== Aangepast rapport ===\n")
        print(aangepaste_rapporttekst)

        ctx["rapport_feedback"] = RapportFeedback(opmerkingen=str(opmerkingen).strip())
        ctx["geaccepteerd_rapport"] = GeaccepteerdRapport(inhoud_markdown=aangepaste_rapporttekst)

Dit is een kleine stap in code, maar een grote stap in controle.

De feedback verdwijnt niet in een chatgeschiedenis. Hij wordt projectdata. Later kunnen we die gebruiken om de rapportgenerator te evalueren en verbeteren.

Segment registreren

De workflow bestaat nu uit zes steps:

VoerParametersIn
LaadGeometrie
BerekenStabiliteit
GenereerRapport
VraagRapportFeedback

from ari_core.workflow.registry import SegmentRegistry

registry = SegmentRegistry()

registry.register(
    Segment(
        name="stabiliteitsrapportage",
        description="Voer grondgegevens in, bereken stabiliteit en rond een rapport af",

            VoerGrondprofielInStep,
            VoerParametersInStep,
            LaadGeometrieStep,

            VraagRapportFeedbackStep,

        requires=[],
    )
)

Deel 2 — Voorbij de eerste demo

De eerste workflow werkt, maar hij is nog beperkt.

Het grondprofiel wordt handmatig ingevoerd. De geometrie wordt als puntenlijst ingevoerd. De rapportgenerator schrijft tekst, maar leert nog niet van correcties.

Dat is normaal voor een eerste versie.

De volgende stap is niet om alles opnieuw te bouwen. De volgende stap is gericht verbeteren waar de workflow in de praktijk schuurt.

In dit voorbeeld doen we drie uitbreidingen:

1. CPT uploaden en interpreteren
2. Custom UI gebruiken voor visuele controle
3. Rapportfeedback opslaan als eval-data

Uitbreiding 1 — CPT uploaden en interpreteren

In veel projecten begint het grondprofiel niet als handmatige invoer, maar als sondering.

Daarom vervangen we de eenvoudige invoerstap door een uploadstap:

CPT uploaden
        ↓
AI of parser maakt eerste interpretatie
        ↓
Ingenieur controleert en corrigeert
        ↓
Geaccepteerd grondprofiel wordt workflowdata

Ook hier geldt hetzelfde principe:

De AI mag een voorstel doen. De ingenieur accepteert de versie die telt.

In deel 1 voert de gebruiker het grondprofiel zelf in. In deel 2 doet de workflow eerst een voorstel. Daarna controleert de ingenieur dat voorstel.

Uitbreiding 3 — Rapportfeedback als eval-data

In de basisworkflow vroegen we feedback op het conceptrapport.

In de geavanceerde workflow gebruiken we die feedback ook om de AI functie beter te maken.

Elke run kan drie versies opleveren:

1. Inputdata
2. AI-conceptrapport
3. Door gebruiker geaccepteerd rapport

Die combinatie is waardevol.

Het is een eval-case.

Een AI engineer kan later controleren:

  • schreef de AI de juiste conclusie?
  • nam hij de veiligheidsfactor correct over?
  • benoemde hij de juiste maatgevende situatie?
  • gebruikte hij de juiste toon voor een civieltechnisch rapport?
  • waren er structurele correcties door gebruikers?

Daarmee verandert feedback in verbeterdata.

Dit is geen training in magische zin. Het is gewoon goede engineering.

Je bewaart voorbeelden waarmee je later kunt testen of een nieuwe prompt, nieuw model of nieuwe rapportgenerator beter werkt dan de vorige versie.

Waarom dit belangrijk is

Veel AI projecten blijven hangen in losse prompts.

Een gebruiker probeert iets. De AI antwoordt. De gebruiker corrigeert. Daarna verdwijnt die correctie.

In een workflow gebeurt iets anders.

De correctie wordt onderdeel van het systeem.

Niet als trainingsdata in vage zin, maar als concrete eval-case:

Bij deze invoer was dit de AI-output.
De gebruiker accepteerde deze versie.
Het verschil tussen die twee is leerzaam.

Dat is hoe je van incidenteel AI gebruik naar structurele verbetering gaat.

Citizen developers en AI engineers

ARI is bruikbaar op twee niveaus.

De domeinexpert kan een eerste workflow bouwen:

  • beschrijf het werkproces
  • laat een AI coding agent de eerste versie maken
  • test lokaal
  • push naar de demo-omgeving
  • laat collega’s feedback geven

Het doel is niet dat iedere civiel ingenieur software engineer wordt.

Het doel is dat domeinexperts hun werkproces concreet genoeg kunnen maken om ermee te experimenteren.

De AI engineer gebruikt daarna de verzamelde data:

  • welke stappen werken goed?
  • waar geven gebruikers correcties?
  • welke rapportpassages worden vaak aangepast?
  • welke prompts falen structureel?
  • welke evals zijn nodig?

Zo ontstaat samenwerking.

De domeinexpert brengt het proces en de feedback. De AI engineer maakt de AI functies betrouwbaarder.

Waarom dit geen chatbot is

Een chatbot kan een rapporttekst schrijven.

Maar een chatbot weet niet vanzelf:

  • welke invoer formeel is geaccepteerd
  • welke berekening deterministisch is uitgevoerd
  • welke output door een ingenieur is goedgekeurd
  • welke correcties als eval moeten worden bewaard
  • welke collega dezelfde workflow later moet kunnen draaien

Daarom is een workflow nodig.

In ARI wordt het proces expliciet:

DataModel
    ↓
Step
    ↓
Segment
    ↓
menselijke verificatie
    ↓
opgeslagen feedback
    ↓
betere AI functie

De waarde zit niet alleen in de eerste AI-output. De waarde zit in de gecontroleerde lus eromheen.

Daarmee wordt ARI niet alleen een manier om AI in een werkproces te gebruiken, maar ook een manier om AI functies gericht te verbeteren.

De beste eerste stap blijft klein.

Kies één werkproces. Bouw één workflow. Laat één collega testen. Verzamel feedback.

Daarna kun je verbeteren.
